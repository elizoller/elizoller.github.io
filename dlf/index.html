DSG: Supporting Digital Scholarship

We are part of the Digital Scholarship Group (DSG), which is a fairly new department in the library.
We consider ourselves an applied research group, responsible for researching, developing, and supporting new avenues for digital scholarship at Northeastern.
A primary goal for the DSG when it was first formed in 2013 was to find a way to support the various projects and research groups that needed to publish digital projects.
Many of our researchers want a simple, customizable site to share project work, and they often look to Wordpress or Omeka to publish and share their content, neither of which is ideal for long term storage.
Other projects turn to the library's Digital Repository Service to store and preserve content, which IS ideal for long term storage, but can't be customized for individual project needs.
Some look to both the DRS for preservation and Wordpress or Omeka for publishing and sharing, duplicating their work across multiple systems.
We wanted to create a tool that would satisfy the preservation AND presentation needs of a project by joining the DRS with customizable web publishing tools,
How do you support project publishing and object storage at your institution?

Planning for the DRS project toolkit started a year ago in October 2014 as a pilot project.
We started with a project team of 4, and have expanded to a team of 10 today, including two developers and 4 research assistants.
At the end of the 2015 pilot, 6 Northeastern projects will be using the Toolkit to present their materials.

Goals

  • Develop a lightweight and sustainable model for preserving and displaying/distributing project materials.
  • Provide Northeastern projects with a low-barrier customized website to share project work.
  • Support Northeastern projects by preserving project materials in the DRS.
  • Encourage collaborative development between Toolkit sites.
In the beginning we had a four goals for the pilot:
We wanted to create a sustainable and shareable digital project environment for scholars.
We wanted to start a program that would scale, and we wanted to encourage project teams to become stewards of their own content, and empower them to customize and develop on their own.
We wanted to avoid providing unique support for individual projects, but work with projects to create customizations that could be utilized by other projects.
And we wanted to make sure that the project materials created were stored securely in the DRS, where the library could guarantee materials would be preserved.

Digital Repository Service DRS

http://repository.library.northeastern.edu
Before I dive into the details, let me give a brief overview of the DRS itself, since it is the foundation of the Toolkit.
The DRS is a homegrown repository, which uses Fedora as the underlying storage architecture and Hydra for the interface.
The interface is clean and focused and its main features are designed to support the widest possible range of material.

Digital Repository Service DRS

http://repository.library.northeastern.edu
DRS users can
  • browse through nested communities and collections or
  • perform basic keyword searching across the full site,
Search results and browse pages allow users to sort results, or refine them using facets.
These are the features we hoped to take advantage of using the Hydra API.

Stakeholder Input

Suggested Features

Before we started work on the Hydra API that would feed DRS data to the Toolkit sites, we wanted to know precisely what functionality was needed to make a good digital project site.
We met in December with representatives from a few DSG projects and partners, including representatives from:
  • the Our Marathon archive
  • the Early Caribbean Digital Archive,
  • University Archives and Special collections,
  • students from public history who work a lot with digital project publishing.
We discussed what they, as digital scholars, need from an integrated digital publishing site like the one we were about to develop.
We identified a few common needs from the group, including:
  • faceted searching,
  • visualizations,
  • sharing,
  • downloading,
  • and galleries or exhibits.
A few features were suggested during this meeting that were worth exploring in the future, but were out of scope for the pilot project, including:
  • two-way data syncing between the publishing tool and the DRS
  • the ability to annotate files
  • XML-aware searching.
We synthesized the outcomes of the stakeholder meeting into a list of features, which would become the first set of tools in the Toolkit.

Toolkit Design

API

  • Search targeted DRS content
  • Browse targeted DRS content
  • Refine search and browse results using facets
  • Sort search Results
  • Brief and full metadata displays
  • Download options

Wordpress

  • Map and timeline visualizations
  • Exhibits and/or galleries
  • Support for non-DRS content
Now that we had a basic spec, we needed to decide what web publishing tools we were willing to support for the Toolkit.
We originally planned on supporting Drupal, Omeka, and Wordpress, but quickly decided that Drupal would be too cumbersome.
Omeka is very popular with digital scholars but we realized that there were few advantages to supporting both Omeka and Wordpress as part of the Toolkit infrastructure during the pilot project,
especially since Wordpress was simpler to customize, and none of our pilot projects expressed a preference for one over the other.
So, to simplify Toolkit development for the pilot, we decided to implement Toolkit sites using Wordpress only.
It's worth noting that the DSG still supports Omeka for other digital projects, just not yet for Toolkit integration.

Workflow

Once the framework was established and the projects were selected, we got to work preparing the project teams for their sites, which involved several steps:
We set up an intake interview to learn more about the project work and to answer any questions the project team may have about the process.
We recommended the project team review their content, with guidance from the universityâ€™s Copyright Officer, if needed.
If the files were not already in the DRS, we helped the projects prepare their files to be ingested, and create metadata, if necessary.
We then generated a Wordpress site from the Toolkit template that Eli will describe, and customized it based on basic project specifications.
We offered training and instruction for project teams throughout the process, though mostly during the file preparation and Wordpress installation stages.
We will review the site prior to launch to make sure it meets some basic Toolkit requirements.
And then we will help the project team promote their site.

DRS API

Written by David Cliff Senior Digital Library Developer
Our Senior Digital Library Developer, David, wrote an API for the DRS. The components we use for the Toolkit are single file retrieval and scoped search.

Single Item

/api/v1/search/[item pid]
Single file retrieval returns JSON for completed, non-embargoed public objects - returns URIs to canonical objects, derivatives, thumbnails, and the metadata in MODS (rails endpoint which reaches out to fedora if the object isnâ€™t already in cache) retrieves MODS, derivatives, and master files

Scoped Search

/api/v1/search/[collection pid]
Scoped search returns JSON for a nested search of a collection. It does some minimal processing around a solr search allowing us to return information about facets and pagination. retrieves Blacklight search results scoped with collection, accepts facets and keywords

DRS + Wordpress

The toolkit itself is a Wordpress plugin and Wordpress child-theme. We chose Wordpress over Omeka because ultimately we thought the built in features of wordpress would be easier to learn and customize by our users. We would be curious to hear feedback on your thoughts and decisions on Wordpress versus Omeka. Our plugin provides templates for three main components of search, browse, and single item view. These templates also enqueue Javascript from the plugin which uses AJAX to reach out to the API, parse the JSON, and present it to users. The plugin provides search, browse, and single item view with extremely minimal Wordpress storage- all Wordpress has in itâ€™s database is the URL for the starting or parent collection in the DRS and some metadata settings. Inputting the collection URL from the DRS associates the collection structure established in the DRS.

Search

Sets up a search page with list style and overrides wordpress search boxes with the search that hits the DRS API. Renders DRS results prominently and wordpress content minimally. Exposes facets and uses AJAX to update the page with fresh DRS results.

Browse

Sets up a browse page with a tiled styling. Hits the DRS API every time to get fresh results

Collection Browse

Allows users to browse by sub-collections of the collection entered in the settings page in a tiled layout

Single Item

Provides a single item page retrieved dynamically from the DRS API with thumbnail or embedded media, MODS, and downloads of derivatives

The other functionality that is a focal point for our users is the ability to create exhibits. Here we mean exhibits to be a page or set of pages that contains narrative and objects together with a variety of different layouts and display options such as galleries. To achieve this inside Wordpress, we take advantage of some of Wordpress' core functionality and plugins developed by other people. We provide a page builder that allows users to create rows and columns for their content. To pull content dynamically from the DRS, we use a set of custom shortcodes. Challenges and Lessons Learned

  • Managing Expectations
  • CSS Customizations
  • Wordpress Media Gallery
  • University Branding
  • Introducing project teams to the idea of repository storage with an API connection to Wordpress and managing pilot expectations was difficult.
    • We did our best to discuss the scope and timeline of the project and establish roles during the intake interview, but additional discussions and documentation was needed.
  • Our beta testing revealed that even experienced Wordpress users were hesitant to alter CSS to customize the site.
    • We compromised by switching to a similar Wordpress theme with more customizations available in the back end.
  • Those who were familiar with Wordpress expected more to be done with the media gallery, or expected that DRS content could be customized using tools similar to those available in the media gallery (like resizing images or adding captions).
    • We tweaked the Wordpress theme to improve customizations for DRS content, and improved documentation to explain when to use the Toolkit media gallery.
  • Establishing how university branding should be incorporated into the site was more difficult than expected.
    • We eventually met with University Marketing to discuss our branding responsibilities. Their rule was, "If it's hosted on University servers, it needs to include University branding.", so we updated the template theme to include basic University branding.
